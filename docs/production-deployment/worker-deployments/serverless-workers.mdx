Serverless Workers let you run Temporal Workers on serverless compute like AWS Lambda.
Deploy your Worker code to a serverless provider, configure a compute provider for the Worker Deployment Version, and Temporal invokes the Worker when tasks arrive.
There are no long-lived processes to provision or scale.

Temporal monitors Task Queues that have a compute provider configured.
When a task arrives and no Worker is polling, Temporal invokes the configured compute target.
The Worker starts, processes available tasks, and shuts down when the invocation window ends.

This guide walks through deploying a serverless Worker on AWS Lambda using the Go SDK.

## Prerequisites {#prerequisites}

- A Temporal Service with the worker controller enabled (Temporal Cloud or self-hosted)
- An AWS account with permissions to create and invoke Lambda functions and create IAM roles
- The [Go SDK](/develop/go) (`go.temporal.io/sdk`)
- The [`aws` CLI](https://aws.amazon.com/cli/)

## Write the Worker code {#write-the-worker-code}

Use the Go SDK's `lambdaworker` package to write a Worker that runs inside a Lambda function.
The package handles the per-invocation lifecycle: connecting to Temporal, polling for tasks, and gracefully shutting down before the Lambda deadline.

```go
package main

import (
	lambdaworker "go.temporal.io/sdk/contrib/aws/lambdaworker"
	"go.temporal.io/sdk/worker"
	"go.temporal.io/sdk/workflow"
)

func main() {
	lambdaworker.RunWorker(worker.WorkerDeploymentVersion{
		DeploymentName: "my-app",
		BuildID:        "build-1",
	}, func(opts *lambdaworker.Options) error {
		opts.TaskQueue = "my-task-queue"

		opts.RegisterWorkflowWithOptions(MyWorkflow, workflow.RegisterOptions{
			VersioningBehavior: workflow.VersioningBehaviorAutoUpgrade,
		})
		opts.RegisterActivity(MyActivity)

		return nil
	})
}
```

Each Workflow must declare a [versioning behavior](/worker-versioning#versioning-behaviors) at registration time, either `AutoUpgrade` or `Pinned`.

For details on configuration options, Lambda-tuned defaults, and the invocation lifecycle, see [Serverless Workers - Go SDK](/develop/go/workers/serverless-workers).

## Deploy the Lambda function {#deploy-the-lambda-function}

Build and deploy the Go binary to AWS Lambda.

### Create the Lambda execution role

Every Lambda function needs an execution role that grants it permission to run.
This is the standard IAM role with `lambda.amazonaws.com` as the trusted principal.

```bash
aws iam create-role \
  --role-name my-temporal-worker-execution \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'

aws iam attach-role-policy \
  --role-name my-temporal-worker-execution \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
```

Grant the execution role whatever additional permissions your Workflow and Activity code needs (S3, DynamoDB, etc.).
This role is separate from the IAM role that Temporal uses to invoke the function.

### Build the binary

Cross-compile for Lambda's Linux runtime:

```bash
GOOS=linux GOARCH=amd64 go build -tags lambda.norpc -o bootstrap ./worker
```

### Configure the Temporal connection

Create a `temporal.toml` file with your Temporal Service connection details.
The `lambdaworker` package reads this file at startup to connect to the Temporal Service.

```toml
[profile.default]
address = "<your-temporal-address>:7233"
namespace = "<your-namespace>"
```

If your Temporal Service requires mTLS authentication, add the TLS configuration:

```toml
[profile.default]
address = "<your-namespace>.<account>.tmprl.cloud:7233"
namespace = "<your-namespace>.<account>"
[profile.default.tls]
client_cert_path = "client.pem"
client_key_path = "client.key"
```

### Bundle and deploy

Package the binary, config file, and any credentials into a zip file:

```bash
zip function.zip bootstrap temporal.toml
```

If using mTLS, include the certificate files:

```bash
zip function.zip bootstrap client.pem client.key temporal.toml
```

Create the Lambda function:

```bash
aws lambda create-function \
  --function-name my-temporal-worker \
  --runtime provided.al2023 \
  --handler bootstrap \
  --architectures x86_64 \
  --role arn:aws:iam::<YOUR_ACCOUNT_ID>:role/my-temporal-worker-execution \
  --zip-file fileb://function.zip \
  --timeout 60 \
  --memory-size 256
```

To update an existing function with new code:

```bash
aws lambda update-function-code \
  --function-name my-temporal-worker \
  --zip-file fileb://function.zip
```

:::note

In production, store credentials in a secret store rather than bundling them with the deployment package.
Encrypt sensitive values using [AWS KMS](https://docs.aws.amazon.com/lambda/latest/dg/configuration-envvars.html#configuration-envvars-encryption).

:::

## Configure IAM for Temporal invocation {#configure-iam}

Temporal needs permission to invoke your Lambda function.
The Temporal server assumes an IAM role in your AWS account to call `lambda:InvokeFunction`.
This requires two things: an IAM role for Temporal to assume, and AWS credentials available to the Temporal server process.

### Create the invocation role

Create a role that allows Temporal to invoke your Lambda function.
The trust policy must include an External ID condition to prevent confused deputy attacks.

```bash
aws iam create-role \
  --role-name temporal-lambda-invoke \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"AWS": "<TEMPORAL_PRINCIPAL_ARN>"},
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "<YOUR_EXTERNAL_ID>"
        }
      }
    }]
  }'
```

Attach a policy that grants permission to invoke and describe the Lambda function:

```bash
aws iam put-role-policy \
  --role-name temporal-lambda-invoke \
  --policy-name invoke-lambda \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Action": ["lambda:InvokeFunction", "lambda:GetFunction"],
      "Resource": "arn:aws:lambda:<REGION>:<ACCOUNT_ID>:function:my-temporal-worker"
    }]
  }'
```

| Value | Description |
|---|---|
| `TEMPORAL_PRINCIPAL_ARN` | The IAM principal that the Temporal server runs as. For Temporal Cloud, this is provided in your Namespace configuration. For self-hosted, this is the IAM identity of the server process (IAM user, EC2 instance role, EKS service account, etc.). |
| `YOUR_EXTERNAL_ID` | A unique identifier that Temporal presents when assuming the role. For Temporal Cloud, this is provided in your Namespace configuration. For self-hosted, this is the value you configure in the compute provider settings. |

### What the External ID prevents

The External ID is a confused deputy protection.
When Temporal assumes your role, it presents your External ID.
A different party cannot use Temporal to access your role because the External IDs would not match.

### Provide AWS credentials to the Temporal server

The Temporal server process needs AWS credentials to assume the invocation role and call Lambda.
The server uses the standard AWS SDK credential chain, so any of these methods work:

- **Temporal Cloud:** No action needed. Temporal Cloud manages credentials internally.
- **EKS:** Use [IAM Roles for Service Accounts (IRSA)](https://docs.aws.amazon.com/eks/latest/userguide/iam-roles-for-service-accounts.html) to associate an IAM role with the Kubernetes service account running the Temporal server.
- **EC2/ECS:** Use an instance role or task role. Credentials are provided automatically through the metadata service.
- **Local development:** Export credentials in the shell before starting the server:
  ```bash
  eval $(aws configure export-credentials --format env)
  ```

The IAM principal used by the server must have permission to call `sts:AssumeRole` on the invocation role you created.

## Register the compute configuration {#register-compute-config}

Create a Worker Deployment Version that links your Lambda function as the compute provider.
The deployment name and build ID must match the values in your Worker code.

The `CreateWorkerDeploymentVersion` API registers a version with a compute configuration.
The compute configuration specifies the provider type (`aws-lambda`), the Lambda function ARN, and the IAM role that Temporal assumes to invoke the function.

```go
// Create the worker deployment
client.CreateWorkerDeployment(ctx, &workflowservice.CreateWorkerDeploymentRequest{
    Namespace:      "default",
    DeploymentName: "my-app",
})

// Create a version with Lambda compute config
client.CreateWorkerDeploymentVersion(ctx, &workflowservice.CreateWorkerDeploymentVersionRequest{
    Namespace: "default",
    DeploymentVersion: &deploymentpb.WorkerDeploymentVersion{
        DeploymentName: "my-app",
        BuildId:        "build-1",
    },
    ComputeConfig: &computepb.ComputeConfig{
        ScalingGroups: map[string]*computepb.ComputeConfigScalingGroup{
            "default": {
                Provider: &computepb.ComputeProvider{
                    Type: "aws-lambda",
                    Details: /* Lambda ARN, role ARN, external ID */,
                },
            },
        },
    },
})
```

Once the CLI supports the serverless flags, registration will be a single command:

```bash
temporal worker deployment create-version \
  --namespace <YOUR_NAMESPACE> \
  --deployment-name my-app \
  --build-id build-1 \
  --aws-lambda-invoke arn:aws:lambda:<REGION>:<ACCOUNT_ID>:function:my-temporal-worker \
  --scaler-min-instances 0 \
  --scaler-max-instances 5
```

## Set the version as current {#set-current-version}

After creating the Worker Deployment Version, set it as the current version.
This step is required. Without it, tasks on the Task Queue will not route to the version, and Temporal will not invoke the Lambda function.

```bash
temporal worker deployment set-current-version \
  --deployment-name my-app \
  --build-id build-1 \
  --ignore-missing-task-queues
```

The `--ignore-missing-task-queues` flag is needed because no Worker has polled the Task Queue yet.
With serverless Workers, the Worker only connects when Temporal invokes the Lambda function.

## Verify the deployment {#verify-the-deployment}

Start a Workflow on the same Task Queue to confirm that Temporal invokes your Lambda Worker.

```bash
temporal workflow start \
  --task-queue my-task-queue \
  --type MyWorkflow \
  --input '"Hello, serverless!"'
```

When the task lands on the Task Queue with no active pollers, Temporal detects the compute provider configuration and invokes your Lambda function.
The Worker starts, connects to Temporal, picks up the task, and processes it.

You can verify the invocation by checking:

- **Temporal UI:** The Workflow execution should show task completions in the event history.
- **AWS CloudWatch Logs:** The Lambda function's log group (`/aws/lambda/my-temporal-worker`) should show invocation logs with the Worker startup, task processing, and graceful shutdown.
